Longtime Programming

Intro

Let's think for a moment about an ATM (automatic teller machine) program where a user is presented with a menu screen asking if they'd like to deposit, withdraw or exit. When the user chooses deposit or withdraw, they're taken to another screen where their transaction is processed, then they click 'OK' button, and they return to the menu screeen. They can do as many transactions as they want, until they choose exit from the menu, and their session ends.

We might write pseudocode for the software like this: 

transactionsDone = false

while(!transactionsDone)
{
	choice = showMenu()
	
	if(choice == deposit)
		runDepositScreen()
	else if(choice == withdraw)
		runWithdrawScreen()
	else if(choice == exit)
		transactionsDone = true
}

The pseudocode makes sense and is understandable, but wait! We can't program the app in this way! Why not? Well, because the deposit and withdrawal screen take a long time to execute. Our JavaScript programming language is based on the idea that each statement takes a very short time to execute and then the program goes promptly to the next step.

That's where Longtime Programming comes in. Longtime programming is an attempt to make pseudocode like the above example translate into real code as closely as possible.
